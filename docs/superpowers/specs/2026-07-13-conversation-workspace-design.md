# 会话化质检与人工待办队列设计

## 目标

降低质检人员在单轮 Trace、低价值问题和重复问题上花费的时间。新增两个互补能力：

1. 将同一买家短时间内的 Trace 聚合为可复盘的多轮会话。
2. 将需要人工判断或立即整改的问题集中为有明确处理建议的待办队列。

本阶段只改变分析结果与展示方式，不删除原始 Trace，不改变现有规则对单轮高风险话术的判定。

## 范围与边界

- 会话按 `buyerIdRaw`、店铺和 60 分钟无活动间隔切分。
- 没有买家 ID、没有时间戳或不能解析出买家问题的 Trace 保持单轮会话。
- 多轮会话定义为至少两条可解析问答 Trace；单轮仍保留为风险证据，但默认不进入聊天记录参考列表。
- 原始 Trace、问题簇、AI 深度分析输入和现有处理状态保持兼容。
- 待办队列是派生视图，不新增自动修改状态或自动发布话术的行为。

真实样本验证：1601 条含买家问题的 Trace 均含买家 ID 与时间戳。以 60 分钟窗口聚合可得到 559 个会话，其中 295 个为多轮会话，覆盖 1337 条 Trace。

## 方案

### 会话聚合

新增纯函数模块 `111/conversation_sessions.py`：

- 输入：Trace 列表及现有 `parse_conversation`、`extract_trace_identity` 的结果。
- 输出：稳定排序的会话，每个会话带有 `id`、`buyerId`、`startTime`、`endTime`、`turnCount`、`isMultiTurn`、`records`。
- 会话 ID 由店铺、原始买家 ID、会话起始时间和首条 Trace ID 计算哈希，避免依赖截断文本。
- 记录按时间升序排列；同一时间时按 Trace ID 排序，保证重跑结果稳定。

`run_analysis()` 新增 `conversationSessions`，但保留原来的 `qaExamples`。问题簇样本增加会话元数据，以便详情优先加载多轮会话，而不是只取聚类命中的第一条 Trace。

### 聊天记录与问题详情

聊天记录区域改名为“多轮质检会话”：

- 默认仅展示 `isMultiTurn=true` 的会话。
- 提供“包含单轮风险记录”的开关，关闭时单轮不会占用主要浏览空间。
- 每行展示买家、会话轮数、持续时间、命中话题及完整问答序列。
- 搜索同时匹配会话内任意问题、回答、买家和话题。

问题详情按以下顺序选择证据：

1. 同问题簇的多轮会话。
2. 没有多轮会话时，展示单轮风险证据，并明确标记“单轮风险证据”。

### 人工待办队列

在概览与问题工作台之间新增“人工待办”区域，所有数据均从 `issueWorkbench` 派生：

- **立即处理**：高风险且置信度大于等于 85，或高优先级且存在绝对承诺规则。
- **需要复核**：人工反馈为 `needs_review`，或置信度低于 70。
- **批量整改**：中优先级以上、出现次数至少 3、且尚未完成处理的问题。

每条待办只显示问题、原因、推荐动作、影响次数、置信度、当前状态和一个跳转到问题详情的操作。相同问题不会同时显示多次，分组按上述优先级互斥。

## 兼容与失败处理

- 旧分析结果没有 `conversationSessions` 时，页面显示“请重新分析以生成多轮会话”，不会把旧 `qaExamples` 当成多轮会话。
- 会话聚合异常时，本地分析仍返回既有问题工作台；聊天区域显示空状态和可理解的提示，不影响抓取或 AI 分析。
- 解析不到时间、买家或问答的 Trace 不会被错误并入其他会话。

## 测试

- 会话窗口边界：59 分钟合并、61 分钟切分。
- 不同买家、缺少买家 ID、时间相同的稳定排序。
- 单轮和多轮覆盖统计。
- 问题详情优先多轮证据，回退时显示单轮标签。
- 人工待办互斥分流、优先级和状态过滤。
- 旧分析前端空状态与“包含单轮风险记录”切换。
- 浏览器测试：默认多轮视图、展开会话、待办跳转和移动端布局。
