# 本地分析 V2 设计

## 背景

现有本地分析使用关键词、回复长度和固定扣分公式快速识别未回复、弱回复和高风险问题。它运行速度快，但会把买家负向表达当成机器人风险信号，也会因为样本绝对数量较大让大店健康分归零。当前共享库的一个分析样本包含 1994 条 Trace、1024 个问题，其中 879 个被标记为高优先级，说明规则需要从“敏感发现”升级为“可解释、可校准的质检判定”。

本阶段参考目标架构中的质检分析层、质量判定层、证据解释层和人工反馈入口，保持现有抓取、问题工作台和 AI 分析接口兼容。

## 目标

- 将本地规则从散落的条件判断改为独立、可版本化的规则引擎。
- 每个问题返回质量维度、命中规则、置信度和原始证据。
- 使用问题率和风险率计算店铺健康分，避免按绝对问题数量无限扣分。
- 降低短回复、买家负面词和正常转人工场景的误判。
- 支持质检人员标记“判断正确、误判、需要复核”，为后续规则校准留下数据。
- 保持 `issueWorkbench` 现有字段可用，AI 深度分析不需要同步重写。

## 非目标

- 本阶段不接入订单状态、商品主数据、知识库正文或策略版本。
- 本阶段不实现整改草稿发布、版本回滚、历史回放和线上指标监控。
- 本阶段不引入外部向量数据库、模型服务或任务队列。
- 本阶段不重做全部前端页面，只增强问题工作台的解释能力和反馈入口。

## 方案选择

### 方案 A：继续在 `app.py` 增加条件

改动小，但规则、评分、接口和业务继续耦合，无法独立测试或版本化，不采用。

### 方案 B：纯 AI 判定

语义能力强，但成本、延迟、稳定性和可复现性不适合作为本地分析基础，不采用。

### 方案 C：可解释规则引擎加 AI 升级入口

本地规则负责确定性判定和证据输出，低置信度问题后续交给 AI 仲裁。该方案兼顾速度、成本、解释性和演进空间，采用此方案。

## 架构

新增 `111/local_analysis_v2.py`，只包含纯函数和规则定义，不访问 Flask、SQLite 或网络。

模块输入是标准化会话：

```json
{
  "traceId": "trace id",
  "question": "买家问题",
  "answer": "机器人回答",
  "searchContent": "原始会话文本",
  "topic": "主题",
  "record": {}
}
```

模块输出是单条质量判定：

```json
{
  "issueType": "未回复|弱回复|高风险|正常",
  "confidence": 0,
  "confidenceLevel": "高|中|低",
  "qualityDimensions": {
    "relevance": 0,
    "completeness": 0,
    "actionability": 0,
    "safety": 0,
    "resolution": 0
  },
  "ruleHits": [
    {
      "ruleId": "QA-NO-ANSWER-001",
      "ruleVersion": "local-v2.0",
      "name": "无有效回复",
      "severity": "高",
      "weight": 100,
      "evidence": "命中的原文或字段"
    }
  ],
  "reasons": []
}
```

`app.py` 负责解析原始 Trace、调用规则引擎、聚类问题、合并历史处理状态，并组装兼容现有页面和 AI 的 `issueWorkbench`。

## 第一批规则

1. `QA-NO-ANSWER-001`：回答为空、占位文本或没有有效机器人回复。
2. `QA-INCOMPLETE-001`：需要步骤、条件或时效的问题只得到泛化回答。
3. `QA-FALLBACK-001`：使用“不清楚、无法确定、稍后”等兜底语句，且没有下一步动作。
4. `QA-MULTITURN-001`：买家重复追问，最后一轮仍没有解决方案。
5. `QA-SENTIMENT-001`：买家出现投诉或强烈不满，回答没有安抚、动作或人工升级。
6. `QA-PROMISE-001`：回答包含绝对承诺、未经校验的退款或时效承诺。
7. `QA-HANDOFF-001`：只要求联系人工但没有入口、条件或上下文交接。
8. `QA-CONSISTENCY-001`：可从结构化多轮消息中确认前后回答矛盾时命中；数据不足时不猜测。

单独的短回复不再直接判定为问题。“好的、可以、已处理、收到”等确认类短答只有在问题明确要求操作步骤或结果说明时才参与弱回复判定。买家负向词只表示场景风险，不能单独证明机器人回答高风险。

## 评分与置信度

### 质量维度

- 相关性：是否回应买家核心意图。
- 完整性：是否包含必要条件、步骤、时效和限制。
- 可执行性：是否给出明确下一步或操作入口。
- 安全性：是否包含不当承诺、错误流程或风险表达。
- 解决度：是否解决当前问题或正确升级人工。

维度默认 100 分，规则只扣减对应维度。问题类型由最严重的高置信规则决定，不再由买家文本中的单个关键词决定。

### 置信度

置信度表示证据强度，不冒充统计概率：

- 90-100：结构化字段直接证明，例如回答为空。
- 70-89：多个规则证据一致，例如兜底话术且没有动作。
- 50-69：语义或会话结构不完整，需要人工复核。
- 低于 50：不进入默认问题队列，只保留为观察信号。

### 店铺健康分

健康分按比例计算，不按绝对数量无限扣分：

```text
100
- 未回复率 × 35
- 高风险率 × 30
- 弱回复率 × 15
- 多轮未解决率 × 10
- 低可执行性率 × 10
```

每项扣分均封顶。样本少于 20 条时标记为“样本不足”，仍显示分数但不用于跨店排名。

## 问题聚类

聚类键从“主题 + 问题前 32 字”升级为：

```text
主题 + 主要规则族 + 归一化关键词签名
```

使用现有 jieba 分词或本地正则分词，按关键词集合相似度合并同义问法，不引入外部服务。问题 ID 使用稳定哈希，确保重新分析后历史处理状态仍能关联。

## 持久化与反馈

SQLite 新增 `issue_feedback` 表：

- `shop_id`
- `issue_id`
- `verdict`：`correct`、`false_positive`、`needs_review`
- `note`
- `updated_by`
- `updated_at`

新增接口：

- `POST /api/issue-feedback`
- `GET /api/issue-feedback?shopId=...`

反馈不直接改变规则结果，只影响页面状态和后续校准统计，避免用户一次点击永久修改分析规则。

## UI

问题列表增加：

- 置信度标签。
- 规则版本。
- 最主要命中规则。

问题详情增加：

- 五个质量维度。
- 规则命中列表和证据原文。
- “判断正确、误判、需要复核”反馈操作。

旧分析结果缺少新字段时继续按现有方式显示，不出现前端错误。

## 数据流

```text
Trace
→ 标准化问答与会话上下文
→ V2 规则逐条评估
→ 置信度与质量维度
→ 相似问题聚类
→ 优先级与比例健康分
→ issueWorkbench
→ 人工反馈
→ 后续规则校准数据集
```

## 错误处理

- 单条 Trace 格式异常时跳过该条并统计数据质量问题，不中断整店分析。
- 规则异常隔离到单条规则，其他规则继续执行。
- 分析结果写入前保留旧结果，完整生成后再替换。
- 规则版本写入分析结果，便于区分旧分析和新分析。

## 测试

新增黄金样本测试，至少覆盖：

- 空回答必须高置信命中。
- 正常确认类短回复不能仅因长度被判弱回复。
- 买家投诉但机器人给出安抚和明确动作时不能判高风险。
- 兜底话术没有下一步时命中弱回复。
- 绝对承诺命中高风险。
- 正常转人工包含入口和条件时不判弱回复。
- 大样本数量增加但问题率不变时健康分保持接近。
- 重新分析后已有问题状态和人工反馈不丢失。

现有后端、AI 任务和 UI 冒烟测试必须继续通过。

## 验收标准

- 实际店铺重新分析后，高优先级问题不再因单个负向词或短回复大量膨胀。
- 每个问题都可以查看规则、版本、置信度和证据。
- 店铺健康分由问题率决定，不再因为 Trace 数量大直接归零。
- 人工反馈可持久化并在刷新后恢复。
- 旧分析和 AI 深度分析接口保持兼容。
