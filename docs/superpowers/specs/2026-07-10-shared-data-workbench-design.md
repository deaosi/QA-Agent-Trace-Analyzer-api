# Shared Data Quality Workbench Design

## Goal

把 QA Agent Trace Analyzer 调整为多人协作的质检工作台：店铺目录和质检数据共享，用户使用自己的 Cookie 抓取数据；同时重做默认工作流和视觉层级，让用户进入店铺后可以直接处理问题。

## Confirmed Product Rules

- 所有登录用户都可以查看和切换共享店铺。
- 每个用户维护自己的平台 Cookie，保存 Cookie 不覆盖其他用户的 Cookie。
- 所有登录用户都可以使用自己的 Cookie 抓取店铺数据。
- 抓取结果合并到店铺共享数据中。
- 同一店铺内使用 `traceId` 去重，重复记录不产生重复质检样本。
- 问题状态、分析结果、知识卡片和聊天记录对所有用户共享。
- AI 配置继续按用户保存。
- 记录抓取时间和抓取用户，便于追踪数据来源，但不在普通列表中暴露 Cookie 内容。

## Recommended Storage Direction

采用混合存储，保持改动可控：

- 用户 Cookie：继续使用按用户名划分的 JSON 文件。
- 共享店铺、trace、分析结果、问题状态：迁移到 SQLite。
- SQLite 使用事务完成 trace 合并和状态更新。
- trace 表以 `(shop_id, trace_id)` 建立唯一约束。
- 已存在的 JSON 数据在首次启动时导入 SQLite，导入过程幂等。
- 原有 JSON 文件保留为回滚备份，不再作为新请求的主数据源。

### Trace merge rule

对每次抓取结果执行事务合并：

1. 校验店铺 ID 和记录结构。
2. 以 `shop_id + traceId` 查找已有记录。
3. 不存在时插入新记录，并写入 `fetched_at`、`fetched_by`。
4. 已存在时保留已有记录的人工字段，使用新抓取数据补齐或更新原始字段。
5. 提交事务后再更新店铺统计信息。

没有 `traceId` 的记录不参与去重，服务端为其生成稳定哈希 ID，避免同一批次内重复写入。

## Information Architecture

默认工作流为：

`选择店铺 -> 查看概览 -> 筛选问题 -> 查看聊天证据 -> 更新状态/复制话术`

页面导航：

- 概览：健康分、问题数量、风险概览、最近抓取状态和待办问题。
- 问题工作台：问题簇列表、优先级/状态/主题筛选、批量处理。
- 聊天记录：共享 trace 分页查看、时间筛选和导出。
- 知识卡片：待补知识卡片、标准答案和转人工规则。
- AI 分析：按用户配置 API，结果写入共享分析结果。
- 数据管理：当前用户 Cookie、抓取参数、抓取日志和共享数据更新时间。

## UI Direction

页面保留现有蓝色品牌识别，但减少渐变、重复指标卡和长页面堆叠，转为安静、密集、适合重复操作的工作台。

### Desktop layout

```text
┌─────────────────────────────────────────────────────────┐
│ 品牌   共享店铺选择   数据状态/时间范围   抓取  分析  用户 │
├──────────────┬──────────────────────────────────────────┤
│ 概览         │ 当前店铺上下文 + 健康概览                  │
│ 问题工作台   ├──────────────────────────────────────────┤
│ 聊天记录     │ 待处理问题列表、筛选、搜索、批量动作        │
│ 知识卡片     ├──────────────────────────────────────────┤
│ AI 分析      │ 点击问题后从右侧打开聊天证据抽屉            │
│ 数据管理     │                                          │
└──────────────┴──────────────────────────────────────────┘
```

- 顶部固定显示当前店铺，避免用户在长页面中丢失上下文。
- 左侧导航固定，主区域只显示当前工作流内容。
- 问题详情使用右侧抽屉，返回列表时保留筛选和滚动位置。
- Cookie 和 AI 配置移出默认首屏，放到数据管理和 AI 页面。
- 移动端将左侧导航折叠为抽屉，表格转为纵向问题条目。
- 使用蓝色作为主操作色，红色/橙色只代表风险，绿色代表已完成。
- 统一按钮高度、表格密度、状态标签和空状态文案。
- 仅保留少量必要动效，并支持 `prefers-reduced-motion`。

## Security and Runtime Hardening Included

- 缺少生产密钥或管理员密码时拒绝启动，禁止匿名回退和固定默认 Flask 密钥。
- AI Base URL 增加 HTTPS、域名和内网地址校验。
- 分页、抓取页数、请求体和超时参数增加上下限。
- 通用异常响应不直接暴露内部异常字符串。
- Docker 自动同步补齐 Chromium 运行依赖并使用 headless 模式。
- 后台同步状态不再依赖多 worker 进程内全局变量；过渡期使用单 worker，后续可替换为外部任务队列。

## Testing

- 保留现有单元测试并新增 SQLite migration、trace 合并、重复 trace、并发状态更新测试。
- 新增用户 Cookie 互不覆盖测试。
- 新增共享店铺和共享分析结果的 API 集成测试。
- 新增缺少密钥、非法分页、非法 AI URL 和 Docker 配置检查。
- 使用 Playwright 验证桌面和移动端布局、店铺切换、问题抽屉和筛选状态保持。
